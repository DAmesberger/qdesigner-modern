# ADR 0001 — Row-Level Security: fix properly

**Status:** **Complete** (Phase 6 at SHA `0758364`). Infrastructure
shipped in Phase 5 (ADR 0011); enforcement landed in Phase 6 via the
ADR 0012/0013/0014/0015 chain. `qdesigner_app` is the non-superuser
non-BYPASSRLS application role; dual-path policies (`app.user_id` and
`app.session_id`) admit authenticated and anonymous fillout
respectively; admin-table mutations are permissive at the DB layer
with `api/access::*` as the authorization gate; FORCE on the four
still-RLS-bound admin tables is declarative posture. Port-plan
portion superseded by [ADR 0009](0009-rls-author-not-port.md).
Connection-pinning trades per-query connection checkout for per-
request checkout; monitor pool saturation in production. (2026-05-16)

**Decision.** Keep RLS as a defense-in-depth layer alongside application-level access checks in `api/access::*`, and make it actually work. Today the `rls_context` middleware calls `set_config(...)` on the sqlx pool, but sqlx hands out a fresh connection per query so handler queries never see the context; meanwhile the 86 RLS policies live in `apps/server/db/migrations/` which `sqlx::migrate!` never runs.

**Consequences.**
- Phase 2 Task 2.4 ports the policies from `apps/server/db/migrations/` into `apps/server/migrations/` (the directory `sqlx::migrate!("./migrations")` actually runs), then deletes the dead directory.
- Phase 3 Task 3.4 rewrites the `rls_context` middleware to acquire and **pin** a connection for the lifetime of the request (typically by running handler work inside a transaction on that connection) so `set_config(...)` persists across queries.
- The existing `rbac_integration` tests are adapted to run against the live schema and become the regression suite for RLS.
- `api/access::*` checks remain — RLS is defense-in-depth, not the sole gate.
